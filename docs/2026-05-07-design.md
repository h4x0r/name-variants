# name-variants — Design Document

**Date:** 2026-05-07
**Status:** Approved
**Location:** `~/src/name-variants`

---

## 1. Purpose

Multilingual name romanization lookup tables. Maps native script forms to all known romanization variants so that matching engines (veil, edng, search normalization) can treat `陳 / Chen / Chan / Tan` as the same entity.

---

## 2. Design Principles

- **Script form is always the canonical key.** Never a romanization. Romanizations are leaves; the script character/word is the root. This makes the table independent of which romanization convention the user chose.
- **Pure data, zero runtime dependencies.** Just Python dicts. Optional `normalize` extra adds opencc + jaconv for callers who need CJK normalization helpers.
- **One file per language/script.** Easy to audit, easy to extend.
- **Lowercase all romanizations** in the tables. Callers casefold before lookup.
- **Traditional Chinese** reduces to Simplified via opencc before lookup — only Simplified chars are keys.

## 4. Data Structure

Each file exports a single dict: `{canonical_key: [romanization, ...]}`

```python
# All romanizations are lowercase — callers casefold before lookup
CHINESE_SURNAME_VARIANTS: dict[str, list[str]] = {
    "陈": ["chen", "chan", "tan", "chin", "zen"],   # 500+ entries
}

ARABIC_NAME_VARIANTS: dict[str, list[str]] = {
    "محمد": ["muhammad", "mohammed", "mohamed"],     # 500+ entries
}
```

---

## 5. Public API (`name_variants/__init__.py`)

```python
from name_variants import lookup_key, ALL_TABLES

# Returns the canonical key shared by all variants
lookup_key("Chan")   # → "陈"
lookup_key("陳")     # → "陈"  (Traditional → Simplified)
lookup_key("Tan")    # → "陈"
lookup_key("Hui")    # → "许"
lookup_key("Xu")     # → "许"
lookup_key("Park")   # → "박"
lookup_key("Pak")    # → "박"
lookup_key("Nguyen") # → "nguyễn"  (tone form is canonical)
lookup_key("Mohammed") # → "محمد"

# Returns None if not in any table (caller falls back to fuzzy)
lookup_key("Kowalski")  # → None
```

`lookup_key` returns `None` for unknown names — callers should fall back to fuzzy matching. This is a change from the veil plan which returned `normalized_self`; returning `None` is cleaner because it makes the "I know this name" / "I don't know this name" distinction explicit.

## 6. Consumers

- **veil** (`~/src/veil`) — entity cross-script merging in `normalize.py`
- **edng** (`~/src/edng`) — name normalization for search/matching

Both install with:
```toml
[tool.uv.sources]
name-variants = { path = "../name-variants", editable = true }
```

## 8. Testing Strategy

- Unit: each table file — spot-check known equivalences
- Unit: no romanization appears in two different entries within the same table (with documented exceptions for genuine ambiguities like `ng`)
- Unit: `lookup_key()` returns correct canonical for known variants
- Unit: `lookup_key()` returns `None` for unknown names
- Unit: Traditional Chinese reduces to Simplified before lookup
- Property: every value in every table is lowercase
- Property: every key is a non-empty string
